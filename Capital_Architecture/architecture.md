## 2. 微内核架构 (Microkernel Architecture)

AgentRT 采用微内核 (Microkernel) 架构模式，以 **CoreKern** 作为系统核心调度器，以 **Cupola** 作为服务胶囊 (Service Capsule)。

### 2.1 CoreKern — 中央调度器

```
┌─────────────────────────────────────────────────┐
│                   CoreKern                       │
│                                                  │
│   ┌─────────────────────────────────────────┐   │
│   │            CoreLoopThree                │   │
│   │                                         │   │
│   │   ┌──────────┐  ┌──────────┐  ┌──────┐ │   │
│   │   │Cognition │─▶│Planning  │─▶│Action│ │   │
│   │   │ 认知模块  │  │ 规划模块  │  │执行模块│ │   │
│   │   └──────────┘  └──────────┘  └──────┘ │   │
│   │                                         │   │
│   └─────────────────────────────────────────┘   │
│                                                  │
│   ┌──────────┐  ┌──────────────┐  ┌──────────┐  │
│   │Task Queue│  │Context Manager│  │Router    │  │
│   │ 任务队列  │  │ 上下文管理器  │  │ 路由分发  │  │
│   └──────────┘  └──────────────┘  └──────────┘  │
│                                                  │
└─────────────────────────────────────────────────┘
```

- **CoreKern** 负责系统级调度、任务路由、上下文管理，是所有服务请求的中央仲裁者。
- **CoreLoopThree** 是 CoreKern 内部的核心执行循环，实现 Cognition → Planning → Action 三阶段流水线。
- CoreKern 本身不包含业务逻辑，所有领域能力通过 Cupola 可插拔地加载。

### 2.2 Cupola — 服务胶囊

```
┌─────────────────────────────────────────────────┐
│  Cupola（服务胶囊）                               │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  Service Interface (服务接口)              │   │
│  │  - 声明式 API 描述                         │   │
│  │  - 版本化协议                              │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  Lifecycle Hooks (生命周期钩子)            │   │
│  │  - init() / start() / stop() / destroy()  │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  Resource Sandbox (资源沙箱)               │   │
│  │  - CPU / Memory / I/O 配额                │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
└─────────────────────────────────────────────────┘
```

每个 Cupola 是一个独立的服务胶囊，具备以下特性：
- **隔离性**：每个 Cupola 运行在独立的资源沙箱中，故障不会扩散。
- **可插拔**：支持热加载和热卸载，无需重启 CoreKern。
- **声明式注册**：通过服务描述符向 CoreKern 注册自身能力。

### 4.1 层间交互机制

- **写入路径 (Bottom-Up)**：原始数据从 L1 进入，经特征提取引擎 (Feature Extractor) 逐层向上抽象，最终形成 L4 模式知识。
- **检索路径 (Top-Down)**：查询从 L4 模式匹配开始，逐层向下细化，直至 L1 原始数据，确保检索的精度和召回率。
- **异步索引**：L1→L2→L3→L4 的抽象过程为异步执行，不阻塞主请求链路。

### 7.1 通信模式选择

| 模式     | 适用场景                     | 延迟       | 可靠性   |
|----------|------------------------------|------------|----------|
| Pub/Sub  | 一对多广播、事件通知          | 低         | 尽力投递  |
| RPC      | 请求-响应、同步调用           | 极低       | 高 (ACK) |
| Stream   | 大数据量、流式传输            | 中         | 高 (背压) |
| Event    | 异步事件、状态变更通知        | 低         | 尽力投递  |

### 8.1 安全原则

1. **默认拒绝 (Deny by Default)**：所有 Cupola 默认最小权限，需显式声明所需能力。
2. **纵深防御 (Defense in Depth)**：四层安全模型层层递进，任一层次被突破仍有后续防护。
3. **最小权限 (Least Privilege)**：每个 Cupola 仅获得完成其职责所需的最小权限集。
4. **不可抵赖 (Non-Repudiation)**：所有操作全量审计日志，支持事后追溯。
